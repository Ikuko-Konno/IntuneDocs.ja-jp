# <a name="endpoint-protection-settings-for-windows-10-and-later-in-intune"></a>Intune での Windows 10 (以降) 用の Endpoint Protection 設定

[!INCLUDE [azure_portal](./includes/azure_portal.md)]

Endpoint Protection プロファイルを使用すると、Windows 10 デバイス上で BitLocker や Windows Defender のようなセキュリティ機能を制御することができます。

この記事では、Endpoint Protection プロファイルの作成について説明します。 Windows Defender ウイルス対策の構成については、[Windows 10 のデバイス制限](device-restrictions-windows-10.md#windows-defender-antivirus)に関するページをご覧ください。 

## <a name="windows-defender-application-guard"></a>Windows Defender Application Guard

次のエディションの Windows 10 でサポートされています。

- Enterprise 
- プロフェッショナル

Microsoft Edge を使用しているときは、Windows Defender Application Guard が組織で信頼されていないサイトから環境を保護します。 分離ネットワーク境界の一覧に含まれないサイトにユーザーがアクセスすると、サイトは Hyper-V の仮想ブラウズ セッションで開かれます。 信頼済みサイトはネットワーク境界によって定義され、デバイスの構成で構成できます。

Application Guard は Windows 10 (64 ビット) デバイスでのみ使用可能です。 このプロファイルを使用すると、Application Guard をアクティブ化するための Win32 コンポーネントがインストールされます。

- **[Application Guard]**:この機能を**有効**にすると、未承認のサイトを Hyper-V 仮想ブラウズ コンテナーで開きます。 **[未構成]** (既定) の場合、(承認済みと未承認を問わず) あらゆるサイトをデバイスで開きます。
- **[クリップボードの動作]**: ローカル PC と Application Guard 仮想ブラウザー間で許可されるコピー/貼り付け操作を選択します。
- **[エンタープライズ サイトの外部コンテンツ]**: 未承認の Web サイトのコンテンツの読み込みを**ブロック**します。 **[未構成]** (既定) の場合、エンタープライズ サイトではないサイトをデバイスで開きます。
- **[仮想ブラウザーから印刷]**: **許可**すると、PDF、XPS、ローカル プリンター、ネットワーク プリンターで仮想ブラウザーからコンテンツを印刷できます。 **[未構成]** (既定) の場合、すべての印刷機能が無効になります。
- **[ログの収集]**: **許可**すると、Application Guard ブラウズ セッション内で発生するイベントのログを収集します。 **[未構成]** (既定) の場合、ブラウズ セッション内でログは収集されません。
- **[ユーザーが生成したブラウザー データを保持する]**: **許可**すると、Application Guard 仮想ブラウズ セッション中に作成されるユーザー データ (パスワード、お気に入り、Cookie など) を保存します。 **[未構成]** (既定) の場合、デバイスを再起動するか、ユーザーがサインアウトすると、ユーザーがダウンロードしたファイルやデータが破棄されます。
- **[グラフィック アクセラレータ]**: **有効に**すると、仮想 GPU にアクセスすることで、グラフィックを大量に利用する Web サイトや動画の読み込みが速くなります。 **[未構成]** (既定) の場合、グラフィックスにデバイスの CPU が使用されます。仮想 GPU は使用されません。
- **[ホスト ファイル システムにファイルをダウンロードする]**: **有効に**すると、ユーザーは仮想ブラウザーからホスト オペレーティング システムにファイルをダウンロードできます。 **[未構成]** (既定) の場合、デバイスにローカルでファイルが保存されます。ホスト ファイル システムにファイルがダウンロードされることはありません。

## <a name="windows-defender-firewall"></a>Windows Defender ファイアウォール

次のエディションの Windows 10 でサポートされています。
- ホーム
- プロフェッショナル
- Business
- Enterprise
- 教育
- モバイル
- Mobile Enterprise

### <a name="global-settings"></a>グローバル設定

これらの設定はすべてのネットワークの種類に適用されます。

- **[ファイル転送プロトコル]**: **ブロック**すると、ステートフル FTP が無効になります。 **[未構成]** (既定) の場合、ファイアウォールでステートフル FTP フィルタリングを行い、セカンダリ接続を許可します。
- **[セキュリティ アソシエーションが削除されるまでのアイドル時間]**: *n* 秒間、ネットワーク トラフィックが検出されなかった場合は、セキュリティ アソシエーションが削除されます。 アイドル時間を秒単位で入力します。
- **[事前共有キーのエンコード]**: **有効に**すると、UTF-8 を利用して事前共有キーがエンコードされます。 **[未構成]** (既定) の場合、ローカル ストア値が使用されます。
- **[IPsec の除外]**: IPsec から特定のトラフィックを除外するように構成します。次が含まれます。
  - **[近隣ノードは IPv6 ICMP の種類コードを検出する]**
  - **[ICMP]**
  - **[ルーターは IPv6 ICMP の種類コードを検出する]**
  - **[Both IPv4 and IPv6 DHCP network traffic]\(IPv4 と IPv6 DHCP の両方のネットワーク トラフィック\)**
- **[証明書失効リストの検証]**: **[CRL の検証を無効にする]**、**[証明書が失効した場合にのみ CRL の検証を失敗する]**、**[何かのエラーが発生したら CRL の検証を失敗する]** など、証明書失効リストの検証をどのように適用するかを決定します。
- **[状況に応じてキー モジュールごとに認証セットを一致させる]**: **有効に**すると、キー モジュールはそれがサポートしていない認証スイートのみを無視します。 **[未構成]** の場合、認証セットに指定されているすべての認証スイートをサポートしていない場合、キー モジュールは認証セット全体を無視します。
- **[パケット キュー]**: IPsec トンネル ゲートウェイのシナリオで、暗号化された受信とクリア テキストの転送について受信側のソフトウェア スケーリングを有効にする方法を入力します。 この設定により、パケットの順序が保持されます。

### <a name="network-settings"></a>ネットワークの設定

これらの設定は特定のネットワークの種類 (**[ドメイン (社内) ネットワーク]**、**[プライベート (検出可能) ネットワーク]**、**[パブリック (検出不可) ネットワーク]** など) に適用されます。

#### <a name="general-settings"></a>全般設定

- **[Windows Defender ファイアウォール]**: **有効に**すると、ファイアウォールと高度なセキュリティがオンになります。 **[未構成]** (既定) の場合、他のポリシー設定に関係なく、すべてのネットワーク トラフィックが許可されます。
- **[ステルス モード]**: **ブロック**すると、ファイアウォールはステルス モードで動作しません。 ステルス モードをブロックすることで、**[IPsec secured packet exemption]\(IPsec 保護パケットの除外\)** もブロックできます。 **[未構成]** (既定) の場合、ファイアウォールはステルス モードで動作します。調査要求に対する応答が阻止されます。
- **[シールド]**: **ブロック**すると、この機能はオフになります。 **[未構成]** (既定) の場合、この設定は有効になります。 この設定と Windows Defender ファイアウォールがオンになっている場合、他のポリシー設定に関係なく、入ってくるすべてのトラフィックがブロックされます。
- **[マルチキャスト ブロードキャストへのユニキャスト応答]**: **[ブロック]** に設定すると、マルチキャスト ブロードキャストへのユニキャスト応答が無効になります。 通常、マルチキャスト メッセージまたはブロードキャスト メッセージへのユニキャスト応答を受信する必要はありません。 これらの応答は、サービス拒否 (DOS) 攻撃、または既知のライブ コンピューターをプローブしようとしている攻撃者を示す可能性があります。 **[未構成]** (既定) の場合、この設定は有効になります。
- **[着信通知]**: **[ブロック]** に設定すると、アプリがポートでの待機をブロックされたとき、ユーザーへの通知を非表示にします。 **[未構成]** (既定) の場合、この設定が有効になります。アプリがポートでの待機をブロックされたとき、ユーザーに通知が表示される場合があります。
- **[受信接続の既定のアクション]**: **[ブロック]** に設定すると、受信接続に対して既定のファイアウォール アクションが実行されません。 **[未構成]** (既定) に設定すると、受信接続に対して既定のファイアウォール アクションが実行されます。

#### <a name="rule-merging"></a>規則のマージ

- **[ローカル ストアからの Windows Defender ファイアウォール規則の承認された適用]**: **有効に**すると、ローカル ストアにあるファイアウォール規則が認識され、適用されます。 **[未構成]** (既定) の場合、ローカル ストアにある承認済みのアプリケーション ファイアウォール規則が無視され、適用されません。
- **[ローカル ストアからのグローバル ポート Windows Defender ファイアウォール規則]**: **有効に**すると、ローカル ストアのグローバル ポート ファイアウォール規則が認識され、適用されます。 **[未構成]** (既定) の場合、ローカル ストアにあるグローバル ポート ファイアウォール規則が無視され、適用されません。
- **[ローカル ストアからの Windows Defender ファイアウォール規則]**: **有効に**すると、ローカル ストアにあるファイアウォール規則が認識され、適用されます。 **[未構成]** (既定) の場合、ローカル ストアのファイアウォール規則は無視され、適用されません。
- **[ローカル ストアからの IPsec 規則]**: **有効に**すると、スキーマまたは接続セキュリティ規則のバージョンに関係なく、ローカル ストアの接続セキュリティ規則を適用します。 **[未構成]** (既定) の場合、ローカル ストアの接続セキュリティ規則は、スキーマと接続セキュリティ規則のバージョンに関係なく、無視され、適用されません。

## <a name="windows-defender-smartscreen-settings"></a>Windows Defender SmartScreen 設定

Microsoft Edge をインストールした、次のエディションの Windows 10 でサポートされます。
- ホーム
- プロフェッショナル
- Business
- Enterprise
- 教育
- モバイル
- Mobile Enterprise

**設定**:

- **[アプリとファイルの SmartScreen]**: ファイルやアプリの実行のために Windows SmartScreen を**有効に**します。 SmartScreen はクラウドをベースとする、フィッシング/マルウェア対策のためのコンポーネントです。 **[未構成]** (既定) の場合、SmartScreen が無効になります。
- **[未確認ファイルの実行]**: Windows SmartScreen で検証されていないファイルをエンド ユーザーが実行する行為を**禁止**します。 **[未構成]** (既定) の場合、この機能は無効になります。エンド ユーザーは未検証のファイルを実行できます。

## <a name="windows-encryption"></a>Windows の暗号化

### <a name="windows-settings"></a>Windows の設定

次のエディションの Windows 10 でサポートされています。

- プロフェッショナル
- Business
- Enterprise
- 教育
- モバイル
- Mobile Enterprise

**設定**:

- **[デバイスの暗号化]**: デバイスの暗号化を有効にするようにユーザーに**求めます**。 Windows のエディションとシステム構成によっては、次のことがユーザーに求められる場合があります。  
  - 別のプロバイダーの暗号化が有効になっていないことを確認する
  - Bitlocker ドライブ暗号化をオフにし、再びオンにする
    
    別の暗号化方式がアクティブになっている場合に Windows の暗号化を有効にすると、デバイスが不安定になる可能性があります。 
- **[メモリ カードの暗号化 (モバイルのみ)]**: デバイスで使用するリムーバブル メモリ カードの暗号化を**要求します**。 **[未構成]** (既定) の場合、メモリ カードの暗号化は要求されません。暗号化をオンにするようにユーザーに求めることはありません。 この設定は、Windows 10 モバイル デバイスにのみ適用されます。

### <a name="bitlocker-base-settings"></a>BitLocker の基本設定

次のエディションの Windows 10 でサポートされています。

- Enterprise
- 教育
- モバイル
- Mobile Enterprise

基本設定は、すべての種類のデータ ドライブに対するユニバーサル BitLocker の設定です。 これらの設定では、エンド ユーザーがすべての種類のデータ ドライブで変更できるドライブ暗号化タスクまたは構成オプションを管理します。

- **[他のディスクの暗号化に対する警告]**: **[ブロック]** を選択すると、別のディスク暗号化サービスがデバイスに存在する場合の警告プロンプトが無効になります。 **[未構成]** (既定) の場合、警告が表示されます。
- **[暗号化方法の構成]**: オペレーティング システム、データ、リムーバブル ドライブ用の暗号化アルゴリズムを構成するには、この設定を**有効に**します。 **[未構成]** (既定) の場合、BitLocker は XTS-AES 128 ビットを既定の暗号化手法として使用するか、他の設定スクリプトによって指定される暗号化手法を使用します。
  - **[オペレーティング システム ドライブの暗号化]**: オペレーティング システム ドライブ用の暗号化の方法を選択します。 XTS-AES アルゴリズムの使用をお勧めします。
  - **[固定データ ドライブの暗号化]**: 固定 (組み込み) のデータ ドライブ用の暗号化の方法を選択します。 XTS-AES アルゴリズムの使用をお勧めします。
  - **[リムーバブル データドライブの暗号化]**: リムーバブル データ ドライブ用の暗号化の方法を選択します。 Windows 10 を実行していないデバイスでリムーバブル ドライブを使用する場合は、AES-CBC アルゴリズムの使用をお勧めします。

### <a name="bitlocker-os-drive-settings"></a>BitLocker OS ドライブ設定
次のエディションの Windows 10 でサポートされています。

- Enterprise
- 教育
- モバイル
- Mobile Enterprise

これらの設定はオペレーティング システムのデータ ドライブにのみ適用されます。

- **[スタートアップ時の追加認証]**: トラステッド プラットフォーム モジュール (TPM) の使用を含む、コンピューター スタートアップの認証要件を構成するには、**[必須]** を選択します。 **[未構成]** (既定) を選択すると、TPM のあるデバイスで基本オプションのみが構成されます。
  - **[互換性のない TPM チップでの BitLocker]**: **[ブロック]** の場合、互換性のある TPM チップがデバイスにないとき、BitLocker の使用が無効になります。 **[未構成]** の場合、互換性のある TPM チップなしでも BitLocker を使用できます。 BitLocker には、パスワードまたはスタートアップ キーが必要になることがあります。
  - **[互換性のある TPM スタートアップ]**: TPM チップを許可するか、許可しないか、または必須とするかを選びます。
  - **[互換性のある TPM スタートアップ PIN]**: TPM チップ搭載のスタートアップ PIN の使用を許可するか、許可しないか、または必須とするかを選びます。 スタートアップ PIN を有効にするには、エンド ユーザーによる操作が必要です。 
  - **[互換性のある TPM スタートアップ キー]**: TPM チップ搭載のスタートアップ キーの使用を許可するか、許可しないか、または必須とするかを選びます。 スタートアップ キーを有効にするには、エンド ユーザーによる操作が必要です。 
  - **[互換性のある TPM スタートアップ キーと PIN]**: TPM チップ搭載のスタートアップ キーおよび PIN の使用を許可するか、許可しないか、または必須とするかを選びます。 スタートアップ キーとスタートアップ PIN を有効にするには、エンド ユーザーによる操作が必要です。
- **[最小 PIN サイズ]**: TPM スタートアップ PIN の最小長を構成するには、この設定を**有効に**します。 **[未構成]** (既定) の場合、ユーザーは長さが 6 から 20 桁のスタートアップ PIN を構成できます。
  - **[最小文字数]**: スタートアップ PIN に必要な文字数を入力します。有効な値は **4**-**20** の範囲となります。
- **[OS ドライブの回復]**: 必要なスタートアップ情報が利用できない場合に BitLocker で保護されたオペレーティング システム ドライブを回復する方法を制御するには、この設定を**有効に**します。 **[未構成]** (既定) 場合、BitLocker の回復については既定の回復オプションがサポートされます。 既定では、DRA が許可され、回復パスワードと回復キーを含む回復オプションがユーザーによって指定されます。回復情報は AD DS にバックアップされません。
  - **[証明書ベースのデータ回復エージェント]**: **[ブロック]** に設定した場合、BitLocker で保護されている OS ドライブではデータ回復エージェントを使用できません。 **[未構成]** (既定) に設定すると、この設定は有効になります。BitLocker で保護されているオペレーティング システム ドライブでデータ回復エージェントを使用できます。
  - **[ユーザーによる回復パスワードの作成]**: 48 桁の回復パスワードの生成をユーザーに許可するか、必須にするか、許可しないかを選びます。
  - **[ユーザーによる回復キーの作成]**: 256 ビットの回復キーの生成をユーザーに許可するか、必須にするか、許可しないかを選びます。
  - **[BitLocker セットアップ ウィザードの回復オプション]**: **[ブロック]** に設定すると、ユーザーには回復オプションが表示されず、変更できません。 **[未構成]** (既定) に設定した場合、BitLocker をオンにしたとき、ユーザーに回復オプションが表示され、変更できます。
  - **[BitLocker 回復情報を AD DS に保存する]**: **有効**にすると、Azure Active Directory (AAD) に BitLocker 回復情報が保存されます。 **[未構成]** (既定) の場合、回復情報は AAD に保存されません。
  - **[AD DS に保存されている BitLocker 回復情報]**: BitLocker 回復情報のどの部分を Azure AD に格納するかを構成します。 次の中から選択します。
    - **回復パスワードとキー パッケージをバックアップする**
    - **回復パスワードのみバックアップする**
  - **[BitLocker を有効にする前に回復情報を AD DS に保存]**: BitLocker 回復情報が Azure Active Directory に正常に保存されていない限り、ユーザーが BitLocker をオンにするのを阻止するには、この設定を**要求**します。 **[未構成]** (既定) の場合、回復情報が Azure Active Directory に正常に保存されていない場合でも、ユーザーは BitLocker をオンにできます。
- **[プリブート回復メッセージと URL]**: ブート前のキー回復画面に表示されるメッセージと URL を構成するには、この設定を**有効に**します。 **[未構成]** (既定) の場合、この機能が無効になります。
  - **[プリブート回復メッセージ]**: プリブート回復メッセージをユーザーに表示する方法を構成します。 次の中から選択します。
    - **既定の回復メッセージと URL を使用する**
    - **空の回復メッセージと URL を使用する**
    - **カスタム回復メッセージを使用する**
    - **カスタム回復 URL を使用する**

### <a name="bitlocker-fixed-data-drive-settings"></a>BitLocker 固定データ ドライブの設定

次のエディションの Windows 10 でサポートされています。

- Enterprise
- 教育
- モバイル
- Mobile Enterprise

**設定**:

- **[BitLocker によって保護されていない固定データ ドライブへの書き込みアクセス]**: BitLocker で保護されていないデータ ドライブへの読み取り専用アクセスを与えるには、**[ブロック]** に設定します。 **[未構成]** (既定) の場合、BitLocker で保護されていないデータ ドライブに読み取り/書き込みアクセスできます。
- **[固定ドライブの回復]**: 必要なスタートアップ情報がないときに BitLocker で保護されている固定ドライブの回復方法を制御するには、この設定を**有効に**します。 **[未構成]** (既定) の場合、この機能が無効になります。
  - **[データ回復エージェント]**: BitLocker で保護されているドライブのポリシー エディターでデータ回復エージェントの使用を**禁止**します。 **[未構成]** (既定) の場合、BitLocker で保護されている固定ドライブでデータ回復エージェントを使用できます。
  - **[ユーザーによる回復パスワードの作成]**: 48 桁の回復パスワードの生成をユーザーに許可するか、必須にするか、あるいは許可しないかを構成します。  
  - **[ユーザーによる回復キーの作成]**: 256 ビットの回復キーの生成をユーザーに許可するか、必須にするか、あるいは許可しないかを構成します。
  - **[BitLocker セットアップ ウィザードの回復オプション]**: **[ブロック]** に設定すると、ユーザーには回復オプションが表示されず、変更できません。 **[未構成]** (既定) に設定した場合、BitLocker をオンにしたとき、ユーザーに回復オプションが表示され、変更できます。
  - **[BitLocker 回復情報を AD DS に保存する]**: **有効**にすると、Azure Active Directory (AAD) に BitLocker 回復情報が保存されます。 **[未構成]** (既定) の場合、回復情報は AAD に保存されません。
  - **[AD DS への BitLocker 回復情報]**: BitLocker 回復情報のどの部分を Azure Active Directory に格納するかを構成します。 次の中から選択します。
    - **回復パスワードとキー パッケージをバックアップする**
    - **回復パスワードのみバックアップする**
  - **[BitLocker を有効にする前に回復情報を AD DS に保存]**: BitLocker 回復情報が Azure Active Directory に正常に保存されていない限り、ユーザーが BitLocker をオンにするのを阻止するには、この設定を**要求**します。 **[未構成]** (既定) の場合、回復情報が Azure Active Directory に正常に保存されていない場合でも、ユーザーは BitLocker をオンにできます。

  - **[別の組織で構成されたデバイスへの書き込みアクセス]**: **[ブロック]** の場合、別の組織で構成されたデバイスへの書き込みアクセスが許可されます。 **[未構成]** (既定) の場合、書き込みアクセスが拒否されます。

#### <a name="rules-to-prevent-office-macro-threats"></a>Office マクロの脅威を防止するための規則

- **電子メール (Web メール/メール クライアント) からドロップされた実行可能なコンテンツ (exe、dll、ps、js、vbs など) の実行 (例外なし)**

#### <a name="rules-to-protect-against-ransomware"></a>ランサムウェアから保護するための規則
- **高度なランサムウェア防止**

> [!TIP]
> これらの規則の詳細については、「[Reduce attack surfaces with Windows Defender Exploit Guard](https://docs.microsoft.com/windows/security/threat-protection/windows-defender-exploit-guard/attack-surface-reduction-exploit-guard)」 (Windows Defender Exploit Guard で攻撃を回避する) を参照してください。

### <a name="exploit-protection"></a>悪用に対する保護

悪用に対する保護を有効にするには、システムとアプリケーションの軽減策を含む XML ファイルを作成します。 この方法には、次の 2 つがあります。

 1. PowerShell: PowerShell コマンドレットの Get-ProcessMitigation、Set-ProcessMitigation、ConvertTo-ProcessMitigationPolicy の 1 つ以上を使います。 これらのコマンドレットは、軽減策設定を構成し、それらの XML 表現をエクスポートします。

 2. Windows Defender セキュリティ センターの UI: Windows Defender セキュリティ センターで [アプリとブラウザー コントロール] をクリックしてから、結果画面の一番下までスクロールして Exploit Protection を見つけます。 まず、[システム設定] と [プログラム設定] タブを使用して、軽減策設定を構成します。 次に、画面の一番下にある [設定のエクスポート] リンクを見つけ、それらの XML 表現をエクスポートします。

メモリ、制御フロー、ポリシーの制限を構成できる XML ファイルをアップロードすることにより、**悪用に対する保護インターフェイスのユーザー編集**をブロックします。 XML ファイル内の設定を使って、アプリケーションが悪用されるのをブロックできます。 **[未構成]** (既定) の場合、カスタム構成はプッシュされません。

## <a name="windows-defender-credential-guard"></a>Windows Defender Credential Guard

次のエディションの Windows 10 でサポートされています。

- Enterprise

Windows Defender Credential Guard は、資格情報盗難攻撃から保護します。 特権のあるシステム ソフトウェアだけがアクセスできるように、シークレットを分離します。

**[Credential Guard]** の設定には次のものが含まれます。

- **[無効]**: 以前に Credential Guard が **[UEFI ロックなしで有効にする]** オプションで有効になっていた場合、Credential Guard をリモートで無効にします。

- **[UEFI ロックで有効化]**: レジストリ キーまたはグループ ポリシーを使って Credential Guard をリモートで無効にできないようにします。

    > [!NOTE]
    > この設定を使った場合、後で Credential Guard を無効にするときは、グループ ポリシーを **[無効]** に設定する必要があります。 また、UEFI の構成情報を各コンピューターから物理的にクリアします。 UEFI の構成が保持されている限り、Credential Guard は有効になっています。

- **[UEFI ロックなしで有効にする]**: グループ ポリシーを使ってリモートで Credential Guard を無効にできるようにします。 この設定を使うデバイスは、Windows 10 バージョン 1511 以降を実行している必要があります。

Credential Guard を有効にすると、次の必要な機能も有効になります。

- **仮想化ベースのセキュリティ** (VBS): 次の再起動時にオンになります。 仮想化ベースのセキュリティは、Windows ハイパーバイザーを使ってセキュリティ サービスのサポートを提供します。
- **ディレクトリ メモリ アクセスによるセキュア ブート**: セキュア ブートと直接メモリ アクセス (DMA) の保護による VBS が有効になります。 DMA 保護は、ハードウェアのサポートを必要とし、正しく構成されたデバイスでのみ有効にされます。

## <a name="local-device-security-options"></a>ローカル デバイスのセキュリティ オプション

これらのオプションを使って、Windows 10 デバイス上のローカル セキュリティ設定を構成します。

### <a name="accounts"></a>[アカウント]

- **[新しい Microsoft アカウントを追加する]**: **[ブロック]** に設定すると、ユーザーはこのコンピューターに新しい Microsoft アカウントを追加できません。 **[未構成]** (既定) に設定すると、ユーザーはデバイスで Microsoft アカウントを使用できます。
- **[パスワードを使用せずにリモート ログオンする]**: **有効に**すると、デバイスのキーボードを利用し、ローカル アカウントと空のパスワードでサインインできます。 **[未構成]** (既定) の場合、物理デバイス以外の場所からローカル アカウントと空のパスワードでサインインできます。

#### <a name="admin"></a>管理者

- **[ローカル管理者アカウント]**: **[有効]** に設定すると、ローカル管理者アカウントが許可されます。 **[未構成]** (既定) に設定されている場合、ローカル管理者アカウントが無効になります。
- **[管理者アカウントの名前を変更する]**: 管理者アカウントのセキュリティ識別子 (SID) に関連付ける別のアカウント名を定義します。

#### <a name="guest"></a>Guest

- **Guest アカウント**: **[有効]** に設定すると、ローカル Guest アカウントが許可されます。 **[未構成]** (既定) に設定されている場合、ローカル Guest アカウントが無効になります。
- **[Guest アカウント名の変更]**: Guest アカウントのセキュリティ識別子 (SID) に関連付ける別のアカウント名を定義します。

### <a name="devices"></a>[デバイス]

- **[ログオンなしでデバイスを装着解除する]**: **[ブロック]** に設定すると、ユーザーは装着している携帯デバイス本体の取り出しボタンを押すことでデバイスの安全に取り外すことができます。 **[未構成]** (既定) の場合、ユーザーはデバイスにサインインし、デバイスを取り外すアクセス許可を受ける必要があります。
- **[共有プリンターのプリンター ドライバーをインストールする]**: **有効に**すると、ユーザーは共有プリンターに接続する過程でプリンター ドライバーをインストールできます。 **[未構成]** (既定) の場合、管理者だけが共有プリンターに接続する過程でプリンター ドライバーをインストールできます。
- **[CD-ROM へのアクセスをローカル アクティブ ユーザーに制限する]**: **[有効]** の場合、対話的にログインしたユーザーだけが CD-ROM メディアを使用できます。 このポリシーが有効なとき、誰も対話的にログオンしていなければ、ネットワーク経由で CD-ROM にアクセスできます。 **[未構成]** (既定) の場合、誰でも CD-ROM にアクセスできます。
- **[リムーバブル メディアのフォーマットと取り出し]**: リムーバブル NTFS メディアをフォーマットして取り出すことができるユーザーを定義します。
  - **未構成**
  - **Administrators**
  - **Administrators と Power Users**
  - **Administrators と Interactive Users**

### <a name="interactive-logon"></a>対話型ログオン

- **[スクリーン セーバーがアクティブになるまでロック画面の非アクティブ時間 (分)]**: 時間を分単位で入力します。デスクトップの対話型ログイン画面で操作のないままこの時間を経過するとスクリーンセーバーが起動します。
- **[ログオンに Ctrl+Alt+Del が必要]**: **[有効]** に設定すると、Ctrl + Alt + Del キーを押さなくても、ユーザーはサインインできます。 **[未構成]** (既定) の場合、ユーザーが Windows にログオンするには、先に Ctrl + Alt + Del キーを押す必要があります。
- **[スマート カードを取り外したときの動作]**: ログオンしているユーザーのスマート カードがスマート カード リーダーから取り出されたときの動作を決定します。 次のようなオプションがあります。

  - **ワークステーションをロックする**: スマート カードが取り出されると、ワークステーションがロックされます。 このオプションを利用すると、スマート カードを持って場所を離れ、なおかつ、保護されたセッションを維持できます。
  - **ログオフを強制する**: スマート カードが取り出されると、ユーザーは自動的にログオフされます。
  - **[リモート デスクトップ サービス セッションであれば、切断する]**: スマート カードを取り出すと、ユーザーをログオフさせずにセッションを切断します。 このオプションを利用すると、後でスマート カードを挿入し、セッションを再開できます。あるいは、スマート カード リーダーを装備した別のコンピューターにスマート カードを挿入し、セッションを再開できます。もう一度サインインする必要はありません。 セッションがローカルの場合、このポリシーの機能は [ワークステーションをロックする] と同じになります。

    詳しくは、[LocalPoliciesSecurityOptions](https://docs.microsoft.com/windows/client-management/mdm/policy-csp-localpoliciessecurityoptions#localpoliciessecurityoptions-interactivelogon-smartcardremovalbehavior) に関するページをご覧ください。

#### <a name="display"></a>ディスプレイ

- **[ロック画面上のユーザー情報]**: セッションがロックされているときに表示されるユーザー情報を構成します。 構成しないと、ユーザーの表示名、ドメイン、およびユーザー名が表示されます。
  - **未構成**
  - **ユーザーの表示名、ドメイン、ユーザー名**
  - **ユーザーの表示名のみ**
  - **ユーザー情報を表示しない**
- **[最後にサインインしたユーザーを表示しない]**: **有効に**すると、ユーザー名が非表示になります。 **[未構成]** (既定) の場合、ユーザー名が表示されます。
- **[サインイン時にユーザー名を表示しない]**: **有効に**すると、ユーザー名が非表示になります。 **[未構成]** (既定) の場合、ユーザー名が表示されます。
- **[ログインのメッセージ タイトル]**: サインインしようとしているユーザーへのメッセージのタイトルを設定します。
- **[ログインのメッセージ テキスト]**: サインインしようとしているユーザーへのメッセージのテキストを設定します。

### <a name="network-access-and-security"></a>ネットワーク アクセスとセキュリティ

- **[名前付きパイプと共有への匿名アクセス]**: **[未構成]** (既定) の場合、匿名アクセスを共有と名前付きパイプの設定に制限します。 匿名でアクセスできる設定に適用されます。
- **[SAM アカウントの匿名の列挙]**: 匿名ユーザーに SAM アカウントの列挙を**許可**します。 Windows は、匿名ユーザーがドメイン アカウントとネットワーク共有の名前を列挙するのを許可します。
- **[SAM アカウントと共有の匿名の列挙]**: **[未構成]** (既定) の場合、匿名ユーザーがドメイン アカウントとネットワーク共有の名前を列挙できます。 SAM アカウントと共有の匿名の列挙を禁止するには、**[ブロック]** に設定します。
- **[パスワードの変更時に LAN Manager ハッシュ値を保存する]**: これを選択すると、次にパスワードを変更するとき、新しいパスワードのハッシュ値を保存することが LAN Manager (LM) に**許可**されます。 **[未構成]** (既定) に設定すると、ハッシュ値は保存されません。
- **[PKU2U 認証要求]**: デバイスに対するオンライン ID 使用の PKU2U 認証要求を**ブロック**します。 **[未構成]** (既定) の場合、このような要求が許可されます。
- **[ソフトウェア アセット管理へのリモート RPC 接続を制限する]**: SAM へのリモート呼び出しをユーザーとグループに禁止することを、既定のセキュリティ記述子定義言語文字列に**許可**します。 **[未構成]** (既定) の場合、既定のセキュリティ記述子定義言語文字列は、SAM へのリモート呼び出しをユーザーとグループに許可します。
  - **セキュリティ記述子**

### <a name="recovery-console-and-shutdown"></a>回復コンソールとシャットダウン

- **[シャットダウン時に仮想メモリのページ ファイルをクリアする]**: **[有効]** に設定すると、デバイスの電源を切るときに仮想メモリのページ ファイルを消去します。 **[未構成]** の場合、仮想メモリは消去されません。
- **[ログオンせずにシャットダウンする]**: **[ブロック]** の場合、Windows ログオン画面でシャットダウン オプションが非表示になります。 ユーザーはデバイスにサインインし、それからシャットダウンする必要があります。 **[未構成]** (既定) の場合、ユーザーは Windows ログオン画面からデバイスをシャットダウンできます。

### <a name="user-account-control"></a>ユーザー アカウント制御

- **[セキュリティで保護された場所を使用しない UIA 整合性]**: **[有効]** に設定されている場合、ファイル システム内のセキュリティで保護されている場所にあるアプリは、UIAccess 整合性でのみ実行されます。 **[未構成]** (既定) の場合、アプリが置かれているファイル システム内の場所がセキュリティで保護されていない場合でも、アプリを UIAccess 整合性で実行できます。
- **[ファイルおよびレジストリの書き込みエラーをユーザーごとの場所に仮想化する]**: **[ブロック]** に設定されている場合、ファイル システムとレジストリに関して、アプリケーションの書き込みエラーは実行時、ユーザーに定義されている場所にリダイレクトされます。 **[未構成]** (既定) に設定すると、保護されている場所にデータを書き込むアプリケーションは失敗します。
- **[署名および検証された実行可能ファイルのみを昇格]**: **有効に**すると、実行可能ファイルを実行する前に PKI 証明書パス検証が強制されます。 **[未構成]** (既定) の場合、実行ファイルの実行前に PKI 証明書パス検証は強制されません。

#### <a name="uia-elevation-prompt-behavior-settings"></a>[UIA 昇格時のプロンプトの動作] の設定

- **[管理者の昇格時のプロンプト]**: 管理者承認モードの管理者に対する昇格時のプロンプトの動作を定義します。
  - **プロンプトを表示せずに昇格する**
  - **セキュリティで保護されたデスクトップで資格情報を要求する**
  - **セキュリティで保護されたデスクトップで同意を要求する**
  - **資格情報を要求する**
  - **同意を要求する**
  - **構成なし**: 非 Windows バイナリに対して同意を要求します
- **[標準ユーザーの昇格時のプロンプト]**: 標準ユーザーに対する昇格時のプロンプトの動作を定義します。
  - **昇格要求を自動的に拒否する**
  - **セキュリティで保護されたデスクトップで資格情報を要求する**
  - **構成なし**: 資格情報を要求します
- **[昇格時のプロンプトをユーザーの対話型のデスクトップにルーティング]**: **有効に**すると、あらゆる昇格要求が、セキュリティで保護されたデスクトップではなく、対話ユーザーのデスクトップに送られます。 管理者と標準ユーザーに対するプロンプト動作ポリシーの設定が使用されます。 **[未構成]** (既定) の場合、管理者と標準ユーザーのプロンプト動作ポリシーの設定に関係なく、すべての昇格要求がセキュリティで保護されたデスクトップに送られます。
- **[アプリのインストールでの昇格時のプロンプト]**: **[ブロック]** に設定すると、昇格の際、アプリケーション インストール パッケージが検出されず、求められません。 **[未構成]** (既定) に設定すると、アプリケーション インストール パッケージで管理者特権が必要なとき、管理ユーザーの名前とパスワードの入力がユーザーに求められます。
- **[セキュリティで保護されたデスクトップを使用しない UIA 昇格時のプロンプト]**: **有効に**すると、UIAccess アプリで、セキュリティで保護されたデスクトップを使わずに昇格を要求できます。 **[未構成]** (既定) の場合、昇格プロンプトではセキュリティで保護されたデスクトップが使用されます。

#### <a name="admin-approval-mode-settings"></a>[管理者承認モード] の設定

- **[ビルトイン Administrator の管理者承認モード]**: **有効に**すると、ビルトイン Administrator アカウントで管理者承認モードを使用できます。 ある操作で特権の昇格が必要になると、その操作を承認するようにユーザーに求めます。 **[未構成]** (既定) の場合、完全管理者特権ですべてのアプリが実行されます。
- **[すべての管理者を管理者承認モードで実行する]**: **[ブロック]** に設定すると、管理者承認モードと関連するすべての UAC ポリシー設定が無効になります。 **[未構成]** (既定) の場合、管理者承認モードが有効になります。

### <a name="microsoft-network-client"></a>Microsoft ネットワーク クライアント

- **[通信にデジタル署名を行う (サーバーが同意した場合)]**: SMB クライアントが SMB パケット署名をネゴシエートするかどうかを決定します。 **未構成**または有効の場合 (既定)、Microsoft ネットワーク クライアントは、セッションのセットアップ時に SMB パケットの署名を実行するようにサーバーに要求します。 サーバーでパケットの署名が有効になっている場合、パケットの署名がネゴシエートされます。 **無効に**すると、SMB クライアントは SMB パケットの署名をネゴシエートしません。
- **[暗号化されていないパスワードをサード パーティの SMB サーバーに送信する]**: **有効に**すると、サーバー メッセージ ブロック (SMB) リダイレクターは、認証時にパスワードの暗号化をサポートしていない Microsoft 以外の SMB サーバーにプレーンテキストのパスワードを送信できます。 **[未構成]** (既定) の場合、パスワードは暗号化されます。

### <a name="microsoft-network-server"></a>Microsoft ネットワーク サーバー

- **[通信にデジタル署名を行う (クライアントが同意した場合)]**: SMB サーバーが SMB パケットの署名を要求するクライアントと SMB パケットの署名をネゴシエートするかどうかを決定します。 **有効に**すると、Microsoft ネットワーク サーバーは、クライアントから要求されたら、SMB パケットの署名をネゴシエートします。 つまり、クライアントでパケットの署名が有効になっている場合、パケットの署名がネゴシエートされます。 **未構成**または無効 (既定値) のとき、SMB クライアントは SMB パケットの署名をネゴシエートしません。
- **[通信にデジタル署名を行う (常時)]**: SMB サーバー コンポーネントによるパケットの署名が必要かどうかを決定します。 **有効に**すると、Microsoft ネットワーク サーバーは、Microsoft ネットワーク クライアントが SMB パケットの署名に同意しない限り、そのクライアントと通信しません。 **未構成**または無効 (既定) のとき、クライアントとサーバーの間で SMB パケットの署名がネゴシエートされます。
